## 📖 概述

AOCI 索引是一个纯文本工件，通过每个源文件或数据库表对应一个结构化条目，表达整个仓库。每个条目结合一个离散标签（架构坐标）和一段连续语义描述（功能、关系、接口、设计约束）。

该协议针对 LLM 辅助软件工程中的一个具体失效模式：长、低熵上下文中的推理退化。AOCI 通过在任务开始之前将文件级信息按固定 schema 重新组织，使 LLM 无需运行时探索即可定位、理解、修改代码。

> 约 **10 万行 LOC** 的仓库通常压缩为 **600–800 行索引** —— token 压缩比约 **1:100 ~ 1:300**，仓库级信息召回率达 **70% ~ 90%**。

一个典型条目：

```
auth.go[WA9JM]: F:JWT authentication middleware |
R:pkg/jwt,model/user | A:- |
S:extract Bearer token, verify JWT, inject user_id
into gin.Context, API key fallback via SHA256

org_repo.go[PO9NTM]: F:organizational data access |
R:model/org | A:- |
S:closure-table for GetTree/GetAncestors,
MoveNode reconstructs subtree relations

users[U-M-M-GUID]: user primary table,
uuid/username/email unique, password_hash bcrypt,
status, is_superadmin, preferences JSONB
```

方法学、评估与理论分析见[论文](https://arxiv.org/abs/2605.02421)。

## 🚀 AOCI 能做什么

AOCI 支持两种互补的开发模式——均经过生产系统验证。

### 模式一 — 索引优先开发

> *先写索引、再从索引生成代码，构建全新系统。*

开发者用自然语言描述需求；LLM 生成完整的 AOCI 索引作为架构蓝图；代码在每个条目的约束下逐文件生成。架构在**索引层级（数百行）**就被审阅——*先于*任何代码生成。

**已验证能力**：单人端到端构建并维护**约 20 万行 LOC** 量级的生产系统，覆盖 Node.js + React、Go + Gin + Vue 3、Go + React 19 + TypeScript 等技术栈，在持续部署条件下达到 **SonarQube 四 A 标准**（可靠性 · 安全性 · 可维护性 · 覆盖率）。

### 模式二 — 已有仓库的迭代开发

> *把已有代码仓库压缩成索引，然后持续基于索引迭代。*

LLM 读取已有仓库并生成 AOCI 索引。索引成为后续所有功能新增、重构、bug 修复的工作蓝图。每次变更**先**更新索引、再传播到代码——使架构图与实现持续保持对齐。

**已验证能力**：在 10 万+ LOC 量级的生产系统上进行 **8 个月以上**持续迭代开发，索引作为每个开发任务唯一的架构事实来源，跨任务的架构债与回归缺陷率为零。

## 🔓 不依赖任何工具链

AOCI **仅是一种格式规范**。它不依赖于任何特定：

- **LLM 模型** — 任何具备结构化生成与长上下文推理能力的前沿模型均可
- **Agent 工具** — 与 Claude Code、Cursor、Cline、Aider、Copilot 等正交，可叠加使用
- **IDE 或编辑器** — 索引就是纯文本，在任意编辑器中均可工作
- **构建系统、语言、框架** — 协议设计上技术栈无关
- **开发者背景** — 专业工程师或非传统编程背景的领域专家均可使用，无需掌握传统编程基础

索引文件可保存在任何能存放文本文件的位置——本地、Git 仓库、文档系统、或 [aoci.ai](https://aoci.ai) 平台。

## 🧭 端到端使用流程

本节描述从零开始基于 AOCI 协议构建一个完整 Web 系统的标准工作流程。流程在生产环境中已被多次验证，可作为新项目的参考路径。

> 适用场景：开发者尚未拥有运行环境与项目代码，希望从需求构思出发，端到端完成"需求 → 索引 → 代码 → 部署"全过程。

---

### 阶段 1 — 基础设施准备

构建可在线访问的生产系统需要前端、后端、数据库三层服务协同工作，因此需要一台可远程访问的服务器。

#### 1.1 选择云服务商

任意主流云服务商均可：

AWS EC2、GCP Compute Engine、Azure VM、DigitalOcean、Linode、阿里云、腾讯云等

#### 1.2 服务器配置

推荐起步配置：

- **操作系统**：Ubuntu LTS（22.04 或更高），命令兼容性最佳
- **规格**：2 vCPU / 2 GB RAM（学习与小型生产系统足够）
- **存储**：默认 40 GB 系统盘
- **网络**：固定公网 IP；带宽按流量计费（少量预付即可）

不需要购买弹性 IP 或自定义 VPC，保持默认配置即可。

#### 1.3 端口配置

在云控制台的安全组规则中放行以下入站端口：

| 端口 | 协议 | 用途 |
|---|---|---|
| 22 | TCP | SSH 远程登录（通常默认开启） |
| 80 | TCP | HTTP 服务 |
| 443 | TCP | HTTPS 服务 |

#### 1.4 远程登录

通过 SSH 连接服务器。设置 root 密码后用以下命令登录：

```bash
ssh root@<your-server-ip>
```

### 阶段 2 — 环境安装

安装目标系统所需的技术栈。AOCI 协议本身技术栈无关，本节以最常用的 Web 应用栈为例。

#### 2.1 默认参考栈

```
前端层 ─── React
后端层 ─── Go（含 Gin 框架）
数据层 ─── PostgreSQL
代理层 ─── Nginx（反向代理 + SSL 终结）
```

#### 2.2 安装方式

[aoci.ai](https://aoci.ai) 平台提供经过验证的系统级安装命令块——开发者无需逐步配置环境，按以下三步完成：

1. 在平台获取对应技术栈的命令块
2. 整段复制到服务器终端
3. 回车执行，等待自动安装完成

其他技术栈（Node.js + Express、Python + Django、Java + Spring Boot、Rust + Axum 等）可根据需求自行配置。AOCI 协议在不同栈上的工作方式一致。

### 阶段 3 — 模型选择

AOCI 索引的生成质量与基于索引的开发效果由承载模型的**结构化生成稳定性**与**长上下文推理能力**决定。生产实践中观察到的能力分层：

| 层级 | 模型示例 | 实际表现 |
|---|---|---|
| 推荐 | Claude Opus| 索引生成质量最高、长仓库推理稳定、协议遵循严格、首选用于生产 |
| 推荐 | Claude Sonnet| 性价比最高、综合体验佳、推荐日常使用 |
| 可用 | GPT-5.5 系列 | 大多数任务可用，复杂跨模块推理可能需要多轮迭代 |
| 不推荐 | 推理或长上下文能力不足的模型 | 协议遵循不稳定，索引漂移率高，不建议生产使用 |

AOCI 协议本身**不绑定任何模型**。开发者可在以下方式中选择：

- 通过 [aoci.ai](https://aoci.ai) 平台接入自有 API Key
- 在任意支持长上下文的独立模型客户端中使用（如 Claude 官方 Web 客户端、ChatGPT、Google AI Studio 等）

---

### 阶段 4 — 需求文档生成

与所选模型开始对话，明确以下三个问题：

1. **核心定位** — 想做的软件是什么、解决什么问题
2. **目标用户** — 给谁用、典型使用场景
3. **功能范围** — 期望包含哪些功能、有哪些非功能性约束（性能、安全、合规）

让模型基于对话产出一份简明的需求文档，迭代至满意为止。

> **关键实践**：首版需求应聚焦核心功能集，避免一次性覆盖完整产品形态。

合理的首版规模参考：**3–8 个核心业务模块**、覆盖**主要工作流**即可。其他功能在后续迭代中增量添加。

---

### 阶段 5 — 索引生成

将需求文档与一份 AOCI 索引参考示例一并提交给模型，要求生成对应的 AOCI 索引。

#### 5.1 提示词关键要素

提示词应明确：

- **目标技术栈**——后端语言/框架、前端框架、数据库类型
- **文件总数控制**——首次使用建议**控制在 50 个代码文件以内**
- **三段输出格式**——索引头部规则、代码索引、数据库索引

#### 5.2 首次使用建议

> 索引规模与首次开发的成功率直接相关：索引越短、系统越简单，首次开发跑通的概率越高，开发者的反馈循环越快，对协议的理解和信心越容易建立。

建议首次项目采用以下参数：

| 项目维度 | 首次建议值 |
|---|---|
| 代码文件总数 | ≤ 50 个 |
| 数据库表总数 | ≤ 15 张 |
| 业务模块数量 | 3–5 个 |
| 预估 LOC 范围 | 10K – 30K |

---

### 阶段 6 — 索引录入与项目初始化

模型生成的索引按三段结构组织，需要分别录入开发环境。

#### 6.1 三段结构

| 段名 | 内容 |
|---|---|
| 索引头部 | 标签字典定义、维度声明、约定 |
| 代码索引 | 每个源文件对应一条条目 |
| 数据库索引 | 每张表对应一条条目 |

#### 6.2 录入方式

**方式 A：使用 [aoci.ai](https://aoci.ai) 平台（推荐）**

1. 在平台新建项目（项目名任意）
2. 将三段索引分别粘贴到对应区域：索引头部、代码索引、数据库索引
3. 保存——平台自动合成完整索引

平台合成后的完整索引为只读，避免误改导致漂移。日常需要手工维护的仅为三个区域：索引头部、代码索引、数据库索引。

**方式 B：纯文本文件**

将完整索引保存为单一文本文件，文件名任意：`AOCI.md` / `aoci.txt` / `index.aoci` / `system-index.md` 均可。文件可放在仓库根目录或任意位置。

### 阶段 7 — 首次开发循环

进入开发会话，启动**三方循环协作模式**：

```
            ┌────────────────────┐
            │   开发者（你）     │
            │   描述需求/任务    │
            └─────────┬──────────┘
                      │
                      ▼
            ┌────────────────────┐
            │   LLM 模型         │
            │   生成命令或代码   │
            └─────────┬──────────┘
                      │
                      ▼
            ┌────────────────────┐
            │   服务器终端       │
            │   执行并返回结果   │
            └─────────┬──────────┘
                      │
                      └──→ 结果反馈回 LLM
                           （循环至主流程跑通）
```

#### 7.1 循环步骤

1. 开发者描述当前需求或下一步任务
2. 模型生成对应命令、配置或代码
3. 开发者将输出复制到终端执行
4. 终端返回执行结果
5. 开发者将关键结果复制回模型上下文
6. 模型根据反馈生成下一步动作
7. 重复 1–6，直至系统主流程可访问

#### 7.2 平台辅助

[aoci.ai](https://aoci.ai) 平台提供**开发模式对话**——内置 AOCI 协议提示词，使模型生成的指令格式直接可执行，减少手工调整开销。开发模式还会根据当前项目的索引上下文，自动在对话中维护协议遵循。

---

### 阶段 8 — 首次完成验证

主流程跑通（可登录、可访问核心页面、关键工作流完整）后执行最后一轮一致性检查：

1. 将完整索引重新发送给模型
2. 要求模型核对索引中声明的所有文件是否均已生成、所有数据库表是否均已建立
3. 修复遗漏的文件、字段或表

至此，首次开发完成。

---

### 阶段 9 — 日常迭代

后续的 bug 修复与功能新增按以下流程进行：

#### 9.1 开启新对话

> 不要在旧会话中继续。长会话上下文质量退化显著（context rot），且 token 成本随历史长度线性增长。新对话从干净状态开始效率最高。

#### 9.2 注入完整索引

将当前最新的完整索引发送至新会话窗口。

#### 9.3 能力确认

询问模型对当前系统的掌握程度：

```
这是你正在开发的系统的 AOCI 索引。
请评估：你对这套系统的信息掌握程度是百分之多少？
是否可以基于此索引接管后续开发？
```

模型确认后即可进入下一步。

#### 9.4 进入开发循环

提出本轮迭代的需求或 bug 描述，进入阶段 7 描述的三方循环模式。

---

### 阶段 10 — 索引同步维护

> **核心原则**：每次代码变更（功能新增、重构、bug 修复）完成后，必须同步更新索引。变更**先**更新索引、再传播到代码。

索引与代码长期脱节将导致 LLM 基于过时架构图进行推理，引入与生产系统中的真实状态不符的修改。

#### 10.1 维护操作分类

| 变更类型 | 操作 |
|---|---|
| 新增源文件 | 让模型生成对应条目；按目录分组定位插入位置，写入新条目 |
| 修改已有文件 | 让模型生成更新后的条目；替换对应位置的原条目 |
| 删除源文件 | 直接删除对应条目 |
| 新增数据库表 | 让模型生成对应条目；在数据库索引区追加 |
| 修改数据库表结构 | 让模型生成更新后的条目；替换对应位置 |

#### 10.2 索引头部规则修改

通常情况下不要修改索引头部（标签字典、维度定义等）。这些规则定义了整套索引的"语法"，频繁改动会导致已生成条目失效。

只有在以下情况下才考虑修改索引头部：
- 项目引入了新的架构层级或业务模块（需在 A/B 字典中扩充）
- 项目引入了新的关键技术特征（需在 D 字典中扩充）
- 完成了对协议的深度理解，需要调整 token 预算策略

修改头部后，应使用模型同步检查所有受影响条目是否仍然合规。

---

### 阶段 11 — 代码托管

严肃的商业系统应将代码托管至 Git 仓库。GitHub、GitLab、Gitee、Bitbucket、自建 Gitea 等均可。

#### 11.1 初始化与首次推送

```bash
cd /path/to/your/project
git init
git remote add origin https://github.com/<username>/<repo>.git
git add .
git commit -m "Initial commit"
git branch -M main
git push -u origin main
```

#### 11.2 日常迭代后推送

每完成一轮迭代后执行：

```bash
git status              # 查看变更
git add .
git commit -m "本次迭代说明"
git push origin main
git log --oneline -5    # 确认最近提交记录
```

#### 11.3 回滚未提交修改

如本轮修改出现严重问题需要丢弃：

```bash
git reset --hard        # 丢弃所有未提交修改
git clean -fd           # 清理未跟踪的新文件
git status              # 确认工作区干净
```

> ⚠ 该操作**不可恢复**。已修改但未提交的代码将永久丢失。**数据库已发生的变更（建表、字段修改、数据写入）不会被 Git 操作回滚**，需要额外的数据库迁移工具或手工脚本处理。

### 流程总结

整个流程一句话概括：

> 准备服务器 → 安装环境 → 选择模型 → 生成需求文档 → 生成 AOCI 索引 → 录入开发环境 → 进入开发循环 → 主流程跑通 → 持续迭代 → 同步更新索引 → 推送至 Git 仓库。

**核心原则**：索引与代码必须持续同步。索引一旦与代码长期脱节，AOCI 的能力即失效，开发会陷入与传统 LLM 编程相同的失效模式。

---

### 学习路径建议

第一次使用 AOCI 时，建议遵循以下路径：

1. **从小型项目入手** — 选择一个 3–5 个模块、10K–30K LOC 规模的项目作为练习对象。一个简单的 CRUD 系统、博客后台、内部工具均可。
2. **完整跑通一次端到端流程** — 即使是非常简单的系统，也要从阶段 1 走到阶段 11，建立完整的工作流肌肉记忆。
3. **掌握索引同步节奏** — 在小项目中养成"代码变更即索引同步"的习惯，这是后续支撑 20 万行级系统的基础。
4. **再进入正式项目** — 完成 1–2 个小项目后，对协议形成直觉，再启动正式商业项目。

直接以大型项目作为第一次实践容易在中途陷入信心崩塌。AOCI 的能力曲线随经验显著提升——第一个项目跑通后，第二个项目的效率通常会显著提高。

## 🆘 常见问题

#### Q1：模型说系统可以访问了，但浏览器打不开页面？

通常是云服务商的安全组未放行 HTTP/HTTPS 端口。检查云控制台的安全组规则，确认 80 和 443 端口已对外开放。具体配置位置因云服务商而异——可截图当前控制台界面发给模型，让其指引具体操作。

#### Q2：终端返回的内容应该复制哪部分给模型？

按以下情况处理：

| 模型请求类型 | 应复制的内容 |
|---|---|
| 要查看代码（如 `cat <file>`） | 复制本次终端输出的所有内容 |
| 要执行命令并查看结果 | 仅复制**执行结果**（通常带序号的输出段），不要将输入的命令重复复制 |
| 输出过长无法完整复制 | 复制最后 10–30 行，或对终端截图。明确告知模型"输出过长，仅截取末尾部分"，模型会根据情况调整后续策略 |

#### Q3：可以使用推理能力较弱的模型完成 AOCI 开发吗？

理论上可以，但实际效果差距显著。AOCI 对模型的**长上下文推理**和**协议遵循**能力要求较高。能力不足的模型在以下场景容易出问题：

- 索引生成时遗漏文件、错误分类标签
- 跨文件推理时丢失约束关系
- 协议遵循不稳定，输出格式漂移

推荐的最低要求是 Claude Sonnet 4.6 或同等水平的模型。具体选型可在 [aoci.ai](https://aoci.ai) 平台上对比测试后决定。

#### Q4：开发过程中模型回复变得混乱怎么办？

通常是上下文质量退化。处理方式：

1. 将当前最新的完整索引重新发送给模型
2. 如果仍然混乱，新建对话窗口、注入索引、继续开发（参考阶段 9.1–9.3）

#### Q5：索引文件应该保存在哪里？

AOCI 协议不强制规定保存位置。常见做法：

- **仓库根目录**——如 `AOCI.md`、随代码一起版本化
- **独立文档系统**——如 Notion、Confluence、Logseq
- **[aoci.ai](https://aoci.ai) 平台**——自动版本管理、漂移检测

无论保存在何处，关键是确保**索引与代码同步更新**。

#### Q6：终端卡住或完全无响应时怎么处理？

| 状况 | 处理 |
|---|---|
| 终端卡住（光标不动、命令未执行完） | 按 `Ctrl+C` 中断当前进程；如无效尝试 `Ctrl+Z` 挂起 |
| 终端完全无响应 | 关闭窗口，重新建立 SSH 连接 |
| 命令执行报错看不懂 | 完整复制错误输出或截图（包括 stack trace）发给模型分析 |

## 📐 协议

### 文件级条目

```
filename[ABCDE-tag]: F:function | R:relations | A:API | S:synopsis
```

方括号内编码**离散标签层**。冒号后编码**连续语义层**，四个元素用 `|` 分隔。

#### 标签层 — 五个正交维度

| 维度 | 含义 | 编码 |
|---|---|---|
| `A` | 架构层级 | 项目自定义字典中的单字符（例如 H=Handler、S=Service、R=Repository、M=Model、W=Middleware、U=Router） |
| `B` | 业务模块 | 单字符（例如 A=Auth、O=Org、C=Credits） |
| `C` | 重要性 | 取值于 {9, 8, 7, 5, 3, 1}——非均匀六档 |
| `D` | 技术特征（可选） | 单字符串联（例如 J=JWT、R=RBAC、T=Tx、C=Crypto） |
| `E` | 代码规模 | 取值于 {S, M, L, XL}——按行数分档 |

> 示例：`WA9JM` 表示 Middleware 层级、Auth 模块、重要性 9、含 JWT 特征、中等规模的文件。

`A`、`B`、`D` 字典由项目自定义，在索引头部声明。

#### 语义层 — 四个元素

| 元素 | 内容 |
|---|---|
| `F` | 文件的业务角色，一句简短描述 |
| `R` | 逗号分隔的依赖文件名引用 |
| `A` | 暴露的 API 或端点；无则填 `-` |
| `S` | 高熵设计决策：降级逻辑、事务边界、加密方案、反直觉契约、运行时不变式。无法从语法本身推断的信息。 |

> `S` 应作为主要内容载体——它在协议中承载每 token 信息密度最高的内容。

### 数据库表条目

```
table_name[domain-table_type-scale_estimate-features]: 字段级描述
```

四维标签用 `-` 分隔：

- `domain` — 功能领域 (U=User、P=Points、I=Indexing、A=Auditing 等)
- `table_type` — 主表 (M)、关联表 (A)、日志表 (L)、配置表 (C)
- `scale_estimate` — 预期行数：S、M、L、XL
- `features` — 串联的属性标记（GUID、JSONB、UNIQ、SOFT、FK 等）

> 示例：`users[U-M-M-GUID]` 解码为用户域主表，中等规模，使用 GUID 标识符。

冒号后是逗号分隔的字段级描述：列语义、约束、编码选择。

### 索引头部

索引以项目特定的头部开始，声明维度字典与预算分配。推荐结构：

```
# AOCI Index — [项目名]
# Spec version: 1.0
#
# Tag dictionaries:
#   A: H=Handler S=Service R=Repository M=Model W=Middleware U=Router
#   B: A=Auth O=Org R=Role C=Credits I=Index ...
#   D: J=JWT R=RBAC T=Tx C=Crypto W=WS S=SSE ...
#
# Importance levels (C): 9 > 8 > 7 > 5 > 3 > 1
#
# Token budget per entry:
#   C=9 : 80–150 tokens
#   C=7 : 50–80 tokens
#   C<=3: 20–40 tokens
#
# Conventions:
#   - 每个源文件或数据库表对应一个条目
#   - R 使用文件名引用，不绑定版本化接口
#   - S 记录无法从语法推导的信息
```

### 条目独立性

每个条目仅依赖其源文件。条目间引用仅通过 `R` 字段中的文件名提及实现；条目不内嵌其他条目的内容。这允许增量重新生成的成本与变更集而非仓库总大小成正比。
